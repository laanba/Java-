## 三、redis分布式锁Redisson

### 底层实现

需要**setnx方法**获取锁才能进行操作，**lua脚本**保证一致性

```
获取锁：
	SET lock value NX EX 10
释放锁：
	DEL lock
```

### 过期时间

为了确保线程能够完成业务，一般通过业务执行时间预估过期时间，或者给锁续期。

redission在这里引入了**watch dog**机制，在线程占有锁的情况下，会开辟一个线程来监听，每隔释放时间的1/3给线程续期。

```
public void redisLock() throws InterruptionException{
    Rlock lock = RedissonCLient.getLock("lock");
    boolean isLock = lock.tryLock(10,TimeUnit.SECONDS);/这里的10指最大等待时间，这段时间会一直请求获取锁
    if(isLOck){
       try{
            system.out.println("执行业务")；
       }
    }finally{
        	lock.unlock();
    }
}
```

### **可重入性**

同一线程可以重入获取锁。

原理：使用hash结构记录线程id和重入次数，也就是key="lock"，value="线程id，1"，当数字变为0时，这组hash值才会被删除，锁才会真正被释放。

### 主从一致性

redisson不能解决主从一致性，可以用redisson提供的红锁解决，但是性能很低，如果有必要保证数据的一致性，可以采用zookeeper实现的分布式锁

## 四、redis主从集群

### 主从集群

由于单节点的redis处理能力有限，为了处理更多的并发操作，可以搭建主从集群，将写操作布置在主节点上，写操作布置在从节点，一般是一主多从模式。

### 主从同步数据的流程

#### 全量同步

全量同步是指主节点和从节点建立第一次同步，具体步骤如下：

```
1，从节点发起同步请求，携带自己的offset和replication id
2，主节点判断是否第一次连接，即replication id相等？
3，如果是，则同步版本信息，发送主节点的offset和complication id覆盖从节点
4，主节点执行bgsave生成RDB文件发送给从节点
5，从节点清空自己的数据，执行发送来的RDB文件
在主节点执行bgsave的过程中，有其他的请求会以命令的格式会被记录在缓冲区，缓冲区是一个日志文件，最后将这个日志文件发送给从节点，保证主从一致。后续的同步也是根据这个日志文件。（这里就是根据offset的差值判断哪些命令要同步）offset是主从节点维护的自增的整数，记录偏移量
```

#### 增量同步

增量同步是指从节点重启后，需要和主节点保持一致进行同步

```
1，主节点判断是否第一此同步
2，如果不是，则将offset差值之间的日志文件发送给从节点
3，从节点根据日志文件数据同步
```

## 五、reids持久化（备份）

### RDB

1. 主动save（停止所有进程）、bgsave（开启子进程）
2. <u>自动bgsave（根据conf文件隔多久时间备份）</u>

原理：开启子进程fork主进程的页表，读物理空间的数据存入磁盘的RDB文件

特点：记录整个内存，恢复快，但记录不完整，占用CPU和内存

### AOF

记录写操作（会记录同一个数据多次写操作，可能会触发重写）

需要打开，配置存储的周期，可以配置重写的阈值（超过上次的百分比、文件大小）

<u>特点：记录执行命令，很细很完整，但是文件很大</u>
